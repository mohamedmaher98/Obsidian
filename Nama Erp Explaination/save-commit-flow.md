# مسار الحفظ في Nama ERP — شرح مبسط

---

## 🧠 الفكرة الأساسية

لما المستخدم يضغط "حفظ"، مش بيحصل حفظ مباشر في قاعدة البيانات.
الموضوع بيمر على **6 طبقات** كل واحدة فيها مسؤولية محددة.

---

## 🗺️ نظرة عامة سريعة

```
المستخدم يضغط زر الحفظ
        ↓
    Vue.js  →  بيرسل الطلب
        ↓
    Servlet  →  بيستقبل الطلب
        ↓
    GCEntityService  →  بيحضّر البيانات
        ↓
    EntityServiceBase  →  بيتحقق من الصلاحيات
        ↓
    EntityMediator  →  بيحفظ فعلاً
        ↓
    ServicesEntityMediator  →  بيقرر: موافقة ولا تثبيت مباشر؟
```

---

## 📋 الطبقات بالتفصيل البسيط

### 1️⃣ Vue.js — الواجهة

المستخدم عنده زرّين:
- **"حفظ مسودة"** → `saveDraft`
- **"حفظ نهائي"** → `saveAndCommit`

لما يضغط أي زر، Vue بترسل HTTP POST للـ Server بـ اسم العملية + بيانات المستند.

---

### 2️⃣ CRUDServlet — استقبال الطلب

- بيستقبل الطلب من Vue
- بيحوّل الـ JSON القادم من المتصفح لـ **DTO** (كائن جافا)
- بيوجّه الطلب للخدمة المناسبة

---

### 3️⃣ GCEntityServiceImpl — تحضير البيانات

بيعمل كذا حاجة قبل ما يكمل:
- بيحدّث بيانات الجلسة
- بيحدّث المرفقات
- بيحوّل الـ DTO لكائن Domain حقيقي
- بيبعت الطلب للطبقة التالية

---

### 4️⃣ EntityServiceBaseImplementor — التحقق والصلاحيات

> ⚠️ هنا بيحصل الكثير من الفحوصات قبل الحفظ

**لو حفظ مسودة:**
1. هل السجل مُراجَع (revised)؟ → لو آه: ❌ خطأ
2. هل مُثبَّت من قبل؟ → لو آه: ❌ ما ينفعش يحفظ مسودة
3. هل الأبعاد صح؟ (مثلاً: الفرع، المخزن...)
4. هل المستخدم عنده صلاحية؟
5. لو كل حاجة تمام → ✅ يكمل للحفظ الفعلي

**لو حفظ نهائي:**
1. بيحفظ مسودة **أولاً** (نفس المسار فوق)
2. لو المسودة فشلت → ❌ وقف
3. لو نجحت → بيكمل لـ `requestCommit()`

---

### 5️⃣ EntityMediator — الحفظ الفعلي

**حفظ المسودة بيمر على:**
1. `entity.saveDraft()` — كل كيان ممكن يعمل حاجة خاصة هنا
2. التحقق من الحقول المطلوبة
3. حفظ البيانات في `draftData` في قاعدة البيانات
4. تسجيل الحدث في الـ Action History

---

### 5️⃣ب ServicesEntityMediator — قرار التثبيت ⭐

> هنا أهم نقطة قرار في العملية كلها

```
requestCommit() وصل هنا
        ↓
    preCommit()  →  تحقق شامل من البيانات
        ↓
    checkForApproval()  →  هل يحتاج موافقة؟
        ↓
   ┌────────────────────────────────────┐
   │                                    │
 آه، يحتاج موافقة              لا، ما يحتاجش
   │                                    │
   ↓                                    ↓
إنشاء ApprovalCase             commitEntity()
إرجاع "بانتظار موافقة" 🔒      التثبيت النهائي ✅
```

---

### 6️⃣ commitEntity() — التثبيت النهائي

لو وصلنا هنا، معناه موافق على كل حاجة:

1. `preCommit()` — تحقق أخير
2. تطبيق التأثيرات المحاسبية والمخزنية
3. `entity.postCommit()` — كل كيان ممكن يعمل حاجة خاصة بعد التثبيت
4. نقل البيانات من `draftData` → `stableData`
5. تسجيل حدث الـ COMMIT في السجل

---

## 🔑 مفاهيم مهمة تعرفها

| المصطلح | معناه ببساطة |
|---|---|
| **Draft** | نسخة مؤقتة من المستند، لسه مش مؤثّرة على المحاسبة أو المخزون |
| **Commit** | التثبيت النهائي، بعده التأثيرات بتتطبق |
| **DTO** | كائن بيانات بسيط بينتقل بين الطبقات |
| **Domain Object** | الكائن الكامل اللي فيه كل المنطق التجاري |
| **preCommit** | تحقق نهائي قبل التثبيت |
| **postCommit** | أكواد بتتنفذ بعد التثبيت مباشرة |
| **ApprovalCase** | طلب موافقة بيتبعت للمسؤول |
| **Hook** | نقطة في الكود بيقدر كل كيان يخصّص سلوكه فيها |

---

## 💡 الفكرة الذكية في التصميم

كل فاتورة أو أمر شراء أو أي مستند عنده **3 نقاط تخصيص**:

```
saveDraft()   →  لو عايز تعمل حاجة خاصة وقت حفظ المسودة
preCommit()   →  لو عايز تتحقق من حاجة قبل التثبيت
postCommit()  →  لو عايز تعمل حاجة بعد التثبيت
```

الـ Framework بيدير الباقي، وكل كيان يركز على منطقه الخاص بس.
